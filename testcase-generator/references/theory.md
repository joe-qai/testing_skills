# 测试理论详解

## 1. 等价类划分法

### 原理

将输入数据划分为若干等价类，从每类中选取代表性数据测试。同一等价类中的数据对发现错误的效力是等价的。

### 应用方法

#### 步骤
1. **识别输入条件**：找出所有输入项
2. **划分等价类**：
   - 有效等价类：符合规则的输入
   - 无效等价类：违反规则的输入
3. **选择测试数据**：每个等价类选一个代表值
4. **设计用例**：一次测试一个等价类，其他保持有效

#### 应用示例

**测试点**：用户名密码登录

**输入项分析**：
- 用户名：6-20字符，字母数字下划线
- 密码：8-20字符，包含大小写字母和数字

**等价类划分**：

| 输入项 | 等价类 | 代表值 | 预期结果 |
|--------|--------|--------|----------|
| 用户名 | 有效 | test_user123 | 成功 |
| 用户名 | 无效-长度不足 | ab | 失败 |
| 用户名 | 无效-长度超限 | verylongusername12345 | 失败 |
| 用户名 | 无效-特殊字符 | user@name | 失败 |
| 用户名 | 无效-空值 | "" | 失败 |
| 密码 | 有效 | Test1234 | 成功 |
| 密码 | 无效-长度不足 | Test12 | 失败 |
| 密码 | 无效-缺大写 | test1234 | 失败 |
| 密码 | 无效-空值 | "" | 失败 |

**生成原则**：
- 一次只测试一个等价类
- 其他输入项保持有效值
- 确保能定位到具体错误

## 2. 边界值分析法

### 原理

错误往往发生在输入范围的边界上。边界值分析通过测试边界及边界附近的值来发现错误。

### 应用方法

#### 五点法
- **离点-下**：最小值-1
- **上点-下**：最小值
- **内点**：中间值
- **上点-上**：最大值
- **离点-上**：最大值+1

#### 应用示例

**测试点**：用户名长度限制 6-20 字符

| 测试点 | 输入值 | 预期结果 | 说明 |
|--------|--------|----------|------|
| 离点-下 | 5字符 "abcde" | 失败 | 小于最小值 |
| 上点-下 | 6字符 "abcdef" | 成功 | 刚好最小值 |
| 内点 | 13字符 "test_user_123" | 成功 | 中间值 |
| 上点-上 | 20字符 "test_user_1234567890" | 成功 | 刚好最大值 |
| 离点-上 | 21字符 "test_user_12345678901" | 失败 | 大于最大值 |

#### 选择策略
- **闭区间** [a, b]：测试 a-1, a, a+1, b-1, b, b+1
- **开区间** (a, b)：测试 a, a+1, b-1, b
- **半开区间** [a, b)：测试 a-1, a, a+1, b-1, b

## 3. 因果图法

### 原理

将输入条件（因）和输出动作（果）之间的逻辑关系图形化，转换为判定表，生成测试用例。

### 基本概念

#### 输入条件（因）
- 输入数据的各种状态
- 取值：0（不满足）或 1（满足）

#### 输出动作（果）
- 系统的输出或操作
- 取值：0（不发生）或 1（发生）

#### 约束关系
- **E约束**：a和b不能同时为1（互斥）
- **I约束**：a、b、c中至少有一个为1（至少一个）
- **O约束**：a和b必须同时为1或同时为0（同时）
- **R约束**：a为1时b必须为1（要求）

### 应用示例

**测试点**：自动售货机

**输入条件**：
- 投币（C1）：是/否
- 选择饮料（C2）：是/否
- 库存充足（C3）：是/否

**输出动作**：
- 出货（E1）：是/否
- 找零（E2）：是/否
- 提示"售罄"（E3）：是/否

**约束**：
- 未投币不能选择饮料（E约束：C1=0 → C2=0）

**判定表**：

| 用例 | C1投币 | C2选择 | C3库存 | E1出货 | E2找零 | E3售罄 |
|------|--------|--------|--------|--------|--------|--------|
| 1 | 0 | 0 | 0 | 0 | 0 | 0 |
| 2 | 1 | 0 | 0 | 0 | 1 | 0 |
| 3 | 1 | 1 | 0 | 0 | 1 | 1 |
| 4 | 1 | 1 | 1 | 1 | 0 | 0 |

**生成用例**：
1. 不投币，不选择 → 无动作
2. 投币，不选择 → 找零
3. 投币，选择，库存不足 → 找零+提示售罄
4. 投币，选择，库存充足 → 出货

## 4. 场景法/流程分析法

### 原理

基于用户实际业务流程设计测试用例，模拟真实使用场景。

### 流程类型

#### 基本流（Happy Path）
最常见、最理想的业务流程。

#### 备选流（Alternative Flow）
业务流程的正常分支。

#### 异常流（Exception Flow）
业务流程中的错误处理。

### 应用示例

**测试点**：网上购物流程

**流程图**：
```
开始 → 浏览商品 → 加入购物车 → 登录 → 填写地址 → 支付 → 订单完成
                                     ↓
                                 未登录 → 登录 → 继续
                                     ↓
                                 余额不足 → 充值 → 继续
                                     ↓
                                 支付超时 → 订单取消
```

**用例设计**：

| 用例 | 流程类型 | 优先级 | 场景 |
|------|----------|--------|------|
| 1 | 基本流 | P0 | 浏览→加入购物车→已登录→填写地址→支付→成功 |
| 2 | 备选流 | P1 | 浏览→加入购物车→未登录→登录→继续→支付→成功 |
| 3 | 异常流 | P2 | 浏览→加入购物车→已登录→余额不足→充值→继续→支付→成功 |
| 4 | 异常流 | P3 | 浏览→加入购物车→已登录→支付超时→订单取消 |

## 5. 正交试验法

### 原理

使用正交表从大量可能的测试用例中选取少数具有代表性的测试用例，既能覆盖主要组合，又能减少用例数量。

### 适用场景
- 输入参数较多（3个以上）
- 每个参数有多个取值
- 参数之间存在组合关系

### 应用示例

**测试点**：用户注册

**输入参数**：
- 性别：男/女
- 年龄段：18-25/26-35/36-45
- 地区：华北/华南/华东

**正交表 L4(2^3)**：

| 用例 | 性别 | 年龄段 | 地区 |
|------|------|--------|------|
| 1 | 男 | 18-25 | 华北 |
| 2 | 男 | 26-35 | 华南 |
| 3 | 女 | 18-25 | 华东 |
| 4 | 女 | 26-35 | 华北 |

**覆盖范围**：
- 每个参数的每个取值至少出现1次
- 参数之间的主要组合都被覆盖
- 用例数量从 2×3×3=18 减少到 4

## 6. 用例数量判断

### 不强制数量

根据场景复杂度自适应调整，不设固定数量。

### 判断标准

| 因素 | 影响 |
|------|------|
| 输入项数量 | 输入项越多，用例越多 |
| 业务规则复杂度 | 规则越复杂，用例越多 |
| 风险等级 | Critical/High 需要更多用例 |
| 测试关注点 | 关注点越多，用例越多 |

### 经验值参考

| 场景复杂度 | 建议用例数 |
|------------|------------|
| 简单（1-2个输入） | 3-5个 |
| 中等（3-5个输入） | 6-10个 |
| 复杂（6+个输入） | 10-15个 |

---

## 7. 测试策略组合

### 推荐组合

| 测试类型 | 方法 | 优先级 | 覆盖重点 |
|----------|------|--------|----------|
| 正向测试 | 等价类 | P0/P1 | 正常业务流程 |
| 反向测试 | 等价类 | P2 | 异常输入 |
| 边界测试 | 边界值 | P3 | 临界值 |
| 流程测试 | 场景法 | P0/P1 | 端到端流程 |
| 组合测试 | 正交/因果 | P2/P3 | 参数组合 |

### 应用优先级

1. **必须覆盖**：
   - 所有核心功能的正向流程
   - 关键输入的边界值
   - 主要异常场景

2. **建议覆盖**：
   - 次要功能的正向流程
   - 一般异常场景
   - 状态转换

3. **可选覆盖**：
   - 低频场景
   - 极端边界
   - 性能压力
