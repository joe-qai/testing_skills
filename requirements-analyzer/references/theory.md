# 需求分析理论基础

## 一、需求分类

### 1.1 功能需求
- **定义**：系统应做什么
- **包含**：用户操作、系统功能、数据处理
- **提取方法**：识别动词(创建/修改/删除/查询/导出)、分析业务流程

### 1.2 非功能需求
- **性能需求**：响应时间、并发数、吞吐量
- **安全需求**：权限控制、数据加密、审计日志
- **可用性需求**：系统可用性、故障恢复
- **易用性需求**：界面友好、操作便捷
- **兼容性需求**：浏览器、操作系统、设备

### 1.3 业务规则
- **定义**：业务运作的约束和规则
- **包含**：校验规则、审批流程、状态流转
- **提取方法**：识别"必须"、"应该"、"不允许"等关键词

### 1.4 约束条件
- **技术约束**：技术栈、框架、版本
- **业务约束**：时间、成本、资源
- **体验约束**：界面风格、交互方式

---

## 二、需求分析常用方法

### 2.1 功能分解
将系统分解为独立的功能模块，每个模块包含多个功能点

### 2.2 流程分析法
分析业务流程，识别流程节点、参与角色、数据流转

### 2.3 用例分析法
从用户视角分析系统如何被使用

### 2.4 场景分析法
分析用户使用系统的典型场景

## 三、需求优先级

### 3.1 MoSCoW方法
| 优先级 | 含义 | 占比 |
|-------|------|------|
| Must have | 必须实现 | 60% |
| Should have | 应该实现 | 20% |
| Could have | 可以实现 | 15% |
| Won't have | 本期不实现 | 5% |

### 3.2 KANO模型
| 类型 | 说明 |
|-----|------|
| 基本需求 | 必须满足，否则不满 |
| 期望需求 | 满足越多，用户越满意 |
| 兴奋需求 | 超出期望，带来惊喜 |

---

## 四、需求依赖关系

### 4.1 依赖类型
- **功能依赖**：功能A需要功能B先完成
- **数据依赖**：功能A需要功能B的数据
- **系统依赖**：本系统与其他系统的交互

### 4.2 依赖表示
```
[A功能]
    |
    +-- [B功能] (依赖: A功能)
```

## 五、质量检查清单

### 5.1 完整性
- [ ] 功能需求覆盖所有业务流程
- [ ] 非功能需求覆盖所有质量属性
- [ ] 约束条件完整

### 5.2 一致性
- [ ] 需求之间无冲突
- [ ] 需求描述无歧义
- [ ] 依赖关系正确

### 5.3 可行性
- [ ] 技术上可实现
- [ ] 在时间和资源约束内可完成

### 5.4 可测试性
- [ ] 每条需求都有对应的验收标准
- [ ] 需求可量化
